# framework 0.1.0

AgentForge4j's first public open-source release: an embeddable Java framework for governed AI
workflows. Workflows are defined in external configuration (JSON + markdown), never hardcoded,
and the runtime executes that definition faithfully — explicit steps, inspectable run state,
retries, human approval gates, structured tool execution, and a decision-level audit trail.

## Highlights

- **Workflow engine and runtime** — a domain model (`WorkflowDefinition`, `AgentDefinition`,
  `StepDefinition`) and execution engine (`agentforge4j-runtime`) that drives a run's
  `WorkflowState` through explicit step behaviours (AGENT, SPAR, WORKFLOW, INPUT, RESOURCE,
  BRANCH, FAIL, RETRY_PREVIOUS, VALIDATE, ASSIGN_CONTEXT, AGGREGATE), all published through a
  structured, replayable event log.
- **Provider-independent model abstraction** — a `ModelTier` concept (`LITE`, `STANDARD`,
  `POWERFUL`) plus `ServiceLoader`-discovered LLM providers keep workflows independent of any one
  vendor. Ten provider modules ship in this release: OpenAI, Anthropic Claude, Google Gemini,
  Mistral, Ollama, vLLM, AWS Bedrock, Azure OpenAI, an OpenAI-compatible generic client, and a
  deterministic fake provider for offline development and testing.
- **Structured tool execution** — tools are invoked through a typed command and a `ToolProvider`
  SPI, never by parsing free text. Ships an HTTP tool provider and Model Context Protocol (MCP)
  client integration, both governed by the same policy layer.
- **Human-in-the-loop gates** — first-class approval, review, and input gates pause a run,
  capture a decision, and resume it, with no run ever advancing past a gate silently.
- **Governed by default, not by configuration** — the default `ToolPolicy` denies remote-network
  and local-process tools unless explicitly allowed; outbound tool HTTP is screened by an egress
  guard that rejects private, loopback, link-local, and cloud-metadata targets and never follows
  redirects; the reserved internal context namespace cannot be written by LLM-emitted commands;
  and step retry budgets are enforced as hard, atomically-reserved ceilings. See
  [SECURITY.md](../../../SECURITY.md) for the full secure-defaults reference.
- **Workflow testing** — a fake LLM client and a dedicated test kit (`agentforge4j-testkit`) make
  workflow runs deterministic, scriptable, and repeatable in tests, with assertion helpers for run
  status, event sequences, and step-visit counts.
- **A shipped workflow catalog** — `agentforge4j-workflows-catalog` ships two workflows out of the
  box: the Workflow Execution Estimator (analyzes a workflow and produces a token-range estimate
  with a recommendation before it runs) and Agent Creator (turns a freeform idea into a
  deterministically-validated agent bundle behind a human approval gate).
- **13 runnable examples** — a standalone example tree (`agentforge4j-examples`, deliberately
  outside the framework reactor) covering framework assembly (quick-start, human-approval,
  Spring Boot, HTTP tools, MCP), one small workflow per language feature (branch, loop, retry,
  SPAR, human-in-the-loop, resource), and both shipped catalog workflows.
- **Spring Boot integration** — `agentforge4j-spring-boot-starter` auto-configures the runtime
  behind a single injectable `AgentForge4j` bean; everything below the starter stays framework-
  neutral and has no dependency on Spring.
- **The visual Workflow Builder** — a companion React component (`@agentforge4j/workflow-builder-
  react`) for authoring and visualizing workflow definitions, released and versioned separately on
  npm (currently `0.6.1`).
- **Documentation** — a generated documentation site (Docusaurus, built from the framework's own
  Javadoc and Spring configuration metadata) alongside 34 architecture decision records and
  per-module READMEs.

## Known limitations

- **The `COLLECTION` step behaviour is not available in this release.** The workflow schema does
  not accept a `COLLECTION` step type, and the runtime rejects any workflow that would reach one
  at build time — this is a deliberate fail-closed gate, not an oversight: the domain types for a
  multi-actor collection gate exist, but the full behaviour is intentionally incomplete and is
  planned for a 0.2.x release (see [ADR-0014](../../adr/0014-collection-gate-step-behaviour.md),
  status Proposed).
- **Pre-1.0 API and workflow-contract stability.** Per the project's pre-1.0 clean-break policy
  (see [ADR-0013](../../adr/0013-pre-1-0-clean-break-compatibility-policy.md), status Accepted),
  breaking changes to APIs, SPIs, and workflow contracts may occur in any 0.x minor release —
  always listed in that release's own notes. Do not treat 0.1.0 as API-frozen.

## Breaking changes

None. This is the first public release.

## Compatibility

- **Java:** 17 baseline. No Java 21+ APIs appear in any published module's main sources.
- **Workflow `schemaVersion`:** `1` — every workflow definition declares this field, and it is
  required and schema-validated at load time.

## Module overview

| Layer | Modules | Role |
|---|---|---|
| Core / runtime | `agentforge4j-core`, `agentforge4j-runtime`, `agentforge4j-config-loader`, `agentforge4j-schema` | Framework-neutral workflow execution: domain model, state, commands, events. No Spring, no database. |
| LLM providers | `agentforge4j-llm-api`, `agentforge4j-llm`, and the ten provider modules listed above | Model/provider integrations behind a common abstraction, discovered via `ServiceLoader`. |
| Tooling | `agentforge4j-tools-http`, `agentforge4j-mcp` | Tool execution, including HTTP tools and Model Context Protocol support. |
| Bootstrap | `agentforge4j-bootstrap` | Programmatic assembly of the full runtime without requiring Spring. |
| Spring integration | `agentforge4j-spring-boot-starter` | Auto-configuration for Spring Boot applications. |
| Testing | `agentforge4j-llm-fake`, `agentforge4j-testkit` | A fake LLM client plus a dedicated test kit for deterministic, repeatable workflow tests. |
| Workflow catalog | `agentforge4j-workflows-catalog` | The shipped workflow catalog, independently versioned on its own release track. |

## Coordinates

`org.agentforge4j:<artifactId>:0.1.0` on Maven Central — for example:

```xml
<dependency>
  <groupId>org.agentforge4j</groupId>
  <artifactId>agentforge4j-bootstrap</artifactId>
  <version>0.1.0</version>
</dependency>
```

Substitute the artifact id for the module you need (see [Module overview](#module-overview)
above, or the root [README](../../../README.md#architecture) for the complete list). The shipped
workflow catalog (`agentforge4j-workflows-catalog`) and the Workflow Builder
(`@agentforge4j/workflow-builder-react` on npm) release independently, on their own tracks.
